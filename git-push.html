Git & Synchronisation

Comment pousser sur Git
sans casser l'historique

Le cas réel où un git push est refusé, puis corrigé proprement avec un pull, une fusion locale et un nouveau push.

Contexte

Pourquoi le push a été rejeté

Le dépôt local avait bien un commit en avance, mais le dépôt distant contenait aussi de nouveaux commits. Git a donc refusé le push pour éviter d'écraser l'historique déjà présent sur GitHub.

Erreur typique : ! [rejected] main -> main (fetch first). Cela signifie que votre branche locale n'est plus alignée avec origin/main.
Méthode

La séquence exacte utilisée

1

Vérifier l'état local

On commence toujours par confirmer si la branche a des commits en avance et si l'arbre de travail est propre.

bash 
git status
2

Tenter le push

Le premier push permet de confirmer si l'historique local peut partir directement ou si une synchronisation est requise.

bash 
git push origin main
3

Récupérer et fusionner le distant

Comme le serveur avait déjà des commits supplémentaires, la bonne réponse était de les récupérer puis de fusionner proprement dans la branche locale.

bash 
git pull origin main --no-rebase

L'option --no-rebase garde une fusion explicite. C'était le choix le plus sûr ici pour intégrer les changements distants sans réécrire l'historique local.

4

Re-pousser après fusion

Une fois les deux historiques réunis localement, le push devient possible et GitHub accepte la mise à jour.

bash 
git push origin main
Comprendre

À quoi sert chaque commande

git status

Affiche l'état de la branche locale, les fichiers modifiés et le décalage éventuel avec la branche distante.

git push origin main

Envoie vos commits locaux vers GitHub. Il échoue si le distant a déjà avancé et que le push ne peut pas être fait en fast-forward.

git pull origin main --no-rebase

Télécharge les nouveaux commits distants puis crée une fusion locale. C'est ce qui évite d'écraser le travail déjà présent sur le dépôt distant.

git push origin main

Le second push publie la branche locale devenue complète, avec vos commits et ceux qui venaient du distant.

Bon réflexe

Routine recommandée avant chaque push

bash 
git status
git pull origin main --no-rebase
git push origin main

Ce que ça évite

  • Les rejets de push de dernière minute
  • Les écarts silencieux entre local et distant
  • Les mauvaises surprises quand plusieurs machines travaillent sur le même dépôt

À surveiller

  • Les conflits de fusion si les mêmes lignes ont changé
  • Les pushes forcés non maîtrisés
  • Les commits faits directement sur GitHub sans récupération locale derrière
Résumé

Le point clé

Je n'ai pas forcé le dépôt. J'ai d'abord laissé Git me dire que le distant avait avancé, puis j'ai intégré cet historique avec un pull, avant de refaire un push. C'est la méthode propre quand on veut tout publier sans perdre ce qui existe déjà sur GitHub.
